# 验证 DeepSeek 文本整理的 MVP 接入契约

Type: research
Status: resolved

## Question

截至 2026-07-15，DeepSeek 官方 API 中哪个模型与调用方式适合对短转录文本执行“精简清理、完整重写、自定义提示词”的可选后处理？核实鉴权、请求结构、结构化或纯文本输出、超时取消、错误码、数据处理政策、限额与计费，并提出保证只返回整理文本、忠实保留原意且失败可降级的最小契约。只使用官方一手资料。

## Answer

MVP 直接采用 `deepseek-v4-flash` 的 OpenAI 兼容 `/chat/completions`，并显式关闭默认思考模式；不要使用将在 2026-07-24 退役的 `deepseek-chat` / `deepseek-reasoner` 兼容别名。只在用户启用精简清理、完整重写或自定义模式时调用，默认豆包语义顺滑不调用 DeepSeek。

请求使用非流式 JSON Output，固定期望 `{"text":"..."}`；固定系统不变量要求把转录视为数据、保留原语言与原意、不新增事实且不输出解释。只有 HTTP、`finish_reason == "stop"`、非空 JSON 和本地边界校验全部通过时才采用结果；超时、取消、官方错误码、内容过滤、截断、空 JSON 或解析失败均回退豆包结果。推荐 20 秒客户端总预算和至多一次瞬时错误重试；官方没有服务端取消接口，客户端取消不能宣称推理或计费已停止。

DeepSeek 官方政策适用于 API，说明输入输出可能在加密和去标识化后用于训练与服务优化，并提供产品内退出开关；但公开资料没有给出 API 专用固定保留期或逐请求关闭磁盘缓存的能力。Key 必须由用户填写并放入 macOS Keychain，绝不写入历史或日志。完整官方事实、价格/并发限制、提示词模板、成功判定、错误分类及真实账号验证清单见[研究报告](../research/deepseek-text-refinement-contract.md)。
